Customer Flow (Step by Step)

Customer taps "Sizing" in sidebar
Sees a grid of all patterns that have size chart data. Only patterns with entries in cs_pattern_size_charts appear.
Picks a pattern (e.g. "Alba Dress")
Loads the sizing page. Server fetches: pattern name, full size chart (body + finished), and saved measurement profile from Supabase if logged in.
Chooses entry method
Two options: Tape measure (most accurate) or Photo upload (beta — uses MediaPipe).
Enters bust, waist, hip, height
If they have a saved profile, fields are auto-filled. They can edit any value. Optionally clicks "Predict full body profile" to get shoulder, arm, leg predictions from the MLP model.
Taps "Analyze my fit"
Measurements are auto-saved to Supabase measurement_profiles table. The analyzing screen shows a Perplexity-style timeline with live data surfacing.
Backend processes (all in one SSE stream)
Step 1: Deterministic size match (instant)
Step 2: MLP body profile prediction (instant)
Step 3: AI streams narrative recommendation (3-8 seconds)
Results appear as visual components
Size badge, fit cards with progress bars, narrative cards (why this size, adjustments, garment notes), expandable data panels.
Optional: Fine-tune preferences
Customer can set fit preferences (fitted/loose, more room at bust, stretch fabric, etc). The AI re-analyzes with a new prompt that includes the previous recommendation + preferences. Page blurs, modal shows, then scrolls to updated results.
Lock size → Download
Customer locks their size. Downloads single-size PDF (A0/A4/US Letter) or custom-fit pattern (scaled to their exact body).

Deterministic Size Matching

File: src/lib/size-matching.server.ts

Pure math — no AI involved. Runs instantly. Compares customer measurements against every size in the chart.

The scoring formula:

// Weighted sum — bust matters most for tops/dresses
score = bustDiff × 1.5 + waistDiff × 1.0 + hipDiff × 1.2

// Lowest score = best match
// If top 2 scores differ by less than 4 → "between sizes"
⚠️
Body chart vs finished chart: Some patterns (like Alba Dress) have no body chart — only finished garment measurements. In that case, the matcher compares against finished measurements and penalizes negative ease (garment smaller than body) with a 2.5x multiplier. You can't wear a dress 10cm smaller than your bust!

Fit categories

CategoryDiff (chart - body)Meaning
tight< -2cmGarment is smaller than body
exact-2 to +2cmPerfect match
comfortable+2 to +6cmNice ease
loose> +6cmVery roomy

Ease calculation

Ease = finished garment measurement - body measurement

Example: If your bust is 90cm and the finished dress bust at Size M is 92cm → ease = +2cm (snug but comfortable for woven fabric).

If the pattern has both body AND finished charts: ease = finished - body chart value (the designed-in room).

MLP Body Profile Prediction

File: src/lib/body-profile-predictor.server.ts

A neural network that predicts 10 body measurements from just 4 inputs. Pure TypeScript — no Python needed at runtime.

Training data
59,000 subjects NHANES + BodyM + Kaggle + synthetic
Model file
mlp-weights.json 25 MB, loaded once
Architecture
4-layer MLP GELU + BatchNorm
Trained with
PyTorch scripts/train-body-measurement.py

Network architecture:

Input:  [height, bust, waist, hip]  →  4 values

Layer 0:  Linear(4 → 512) → GELU → BatchNorm
Layer 4:  Linear(512 → 1024) → GELU → BatchNorm
Layer 8:  Linear(1024 → 512) → GELU → BatchNorm
Layer 12: Linear(512 → 256) → GELU
Layer 14: Linear(256 → 10) → denormalize

Output: [bust, waist, hip, height, weight, arm_length,
         shoulder, neck, leg_length, arm_circ]  →  10 values

What the customer sees

After entering bust/waist/hip/height, they can click "Predict full body profile" → this calls POST /api/ai/body-profile → returns shoulder, arm length, leg length. These appear as editable fields — the customer can override any prediction before analyzing.

If they don't click predict, the MLP still runs server-side during the streaming analysis.

How Height Is Used (Jovi's Question!)

❓ "Can we see in the back end how it currently calculates taking into account the height of the customer?"
Yes! Height is used in three places:

1. MLP Body Profile Prediction

File: src/lib/body-profile-predictor.server.ts, line 80

Height is the first input to the neural network: [height, bust, waist, hip]. The MLP uses height to predict proportional measurements — a taller person will have longer arms, longer legs, wider shoulders. The model learned these proportions from 59,000 real bodies.

// Input order: [height, bust, waist, hip]
const input = [height_cm, bust_cm, waist_cm, hip_cm];

// Forward pass through 4-layer MLP...
// Output: shoulder, arm_length, leg_length, etc.

2. AI Prompt Context

File: src/routes/api/ai/size-intelligence/stream/+server.ts, buildPrompt()

Height is included in the prompt sent to Claude. The AI uses it to recommend length adjustments. Patterns are typically designed for ~168cm. The AI says things like "At 175cm, you may want to add 2cm to the bodice length" or "At 160cm, consider shortening the skirt by 3cm."

`CUSTOMER: bust 90cm, waist 60cm, hip 90cm, height 169cm`

// AI system prompt says:
"Heights are designed for ~168cm. Adjust lengths proportionally
 for taller/shorter."

3. Finished Garment Length Comparison

The finished chart includes full_length_cm per size. The AI compares this against the customer's height to recommend specific adjustments. Example: "Size M is 137cm long — at your 169cm, this will hit well below the knee."

📌
Height does NOT affect the size recommendation itself — bust/waist/hip determine the size. Height affects length adjustments and body profile predictions only. This is correct — a 160cm person and a 180cm person with the same bust/waist/hip should wear the same size, just with different length alterations.

How re-analysis works

A new API call is made to the same streaming endpoint, but with two extra fields:

preferences — the customer's selections

previous_recommendation — the FULL text of the initial AI response

The AI uses a different system prompt (FOLLOWUP_SYSTEM_PROMPT) that tells it to focus on what changed. It may suggest a different size. The size badge, fit cards, and all components update automatically.

🔄
During re-analysis: the entire page blurs, a modal popup appears with a shimmer progress bar, and when done the page scrolls back to the top showing updated results.

Custom-Fit Pattern Generation

This is the Beta feature!

After locking a size, the customer can request a custom-fit pattern — scaled to their exact body measurements. Available in A0, A4, and US Letter PDF formats.

Calculate grading
pattern-grading.server.ts — finds closest standard size, calculates ease, computes scale factors (width + height). Warns if adjustment exceeds 15% or 25%.
Show preview
Customer sees: original pattern measurements vs their custom measurements, the delta (e.g. +2.3cm bust), and a confidence badge (high/medium/low).
Generate scaled PDFs
pattern-files.server.ts — downloads original A0/A4/US Letter PDFs from Supabase Storage, uses pdf-lib to scale every page with page.scaleContent(scaleW, scaleH).
Download
Customer downloads A0, A4, or US Letter — each individually. The PDF is proportionally scaled to their body + ease.
// Scale formula
scale_width  = custom_bust_finished / sample_bust_finished
scale_height = custom_length / sample_length  (or √scale_width)

// Example: bust 90cm + 4cm ease = 94cm custom finished
// Sample size S finished bust = 88cm
// scale_width = 94 / 88 = 1.068 → 6.8% wider
